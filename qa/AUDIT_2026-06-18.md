## P0 — Runtime bloqueante

### P0-1 · `AdminUsuariosPage.tsx:84` — `ReferenceError: rolesLoaded is not defined`
- **Arquivo:** `src/pages/admin/AdminUsuariosPage.tsx:84`
- **Sintoma:** `enabled: rolesLoaded && Boolean(isAdmin)` referencia variável que NÃO é desestruturada de `useAuth()` (linha 43 só pega `user, isAdmin, isDev`).
- **Impacto:** crash da página `/admin/usuarios` na primeira render (ReferenceError) → admin não acessa gestão de usuários, aprovação de descontos e password-reset.
- **Causa:** patch parcial (comentário `FIX 2026-06-18: rolesLoaded garante JWT pronto`) sem adicionar `rolesLoaded` ao destructuring.
- **Fix proposto:** desestruturar `rolesLoaded` do `useAuth()` (já existe no contexto? confirmar) **OU** trocar gate por `Boolean(user) && Boolean(isAdmin)`.

## P1 — Funcional / segurança UX

### P1-1 · `useProductsLightweight.ts:61,64` — campos fantasma na resolução de categoria
- **Erros TS:** `TS2551` em `leaf_category_id` (sugere `category_id`), `TS2339` em `leaf_category_name`.
- **Impacto:** o `as string | undefined` mascara em runtime → cast retorna `undefined` → cai sempre no fallback `p.category_id || p.main_category_id`. A intenção do fix BUG-D ("preferir a categoria mais profunda") **não está sendo cumprida**. Produtos com hierarquia mostram categoria errada nos cards.
- **Fix proposto:** ou (a) adicionar `leaf_category_id`/`leaf_category_name` ao tipo `LightweightProduct` se a query SQL realmente retorna esses campos, ou (b) remover o pseudo-cast e ficar só com `category_id || main_category_id`.

### P1-2 · `useSupplierSalesRanking.ts:49,64` — RPC fora do types.ts
- **Erro TS:** `TS2345` — `fn_get_product_intelligence_all` não está em `Database["public"]["Functions"]`.
- **Impacto:** se a função existir no banco mas faltar no types.ts → só erro de TS. Se **não existir**, todo o ranking de fornecedores quebra silenciosamente (cai no fallback "MV not populated").
- **Investigação necessária:** confirmar via `SELECT * FROM pg_proc WHERE proname = 'fn_get_product_intelligence_all'`.
- **Fix proposto:** regenerar types.ts (regra do projeto: anotar count antes/depois). Se a função não existir, restaurar o `dbInvoke` anterior ou criar a função (precisa aprovação do PO).

### P1-3 · Toast leaks (+7) — vazamento de erro técnico
- **Arquivos:**
  - `src/components/admin/products/new-supplier/useNewSupplierForm.ts:257,289,519`
  - `src/components/admin/suppliers-manager/useSuppliersManager.ts:508,529,565,600`
- **Padrão:** `toast.error(err instanceof Error ? err.message : 'Erro ao …')`.
- **Impacto:** quebra a SSOT `sanitize-message` — usuário não-dev vê stack traces, `Failed to fetch`, mensagens em UPPER_SNAKE_CASE. Já é runtime-bloqueado pelo `installSafeToast()`, mas o gate static falha (regressão de baseline).
- **Fix proposto:** trocar por copy fixa + `description: sanitizeError(err)` (padrão documentado).

### P1-4 · 4 edge functions sem entrada no `edge-authz-manifest.ts`
- **Funções:** `audit-suite`, `generate-blurhashes`, `hash-product-images`, `word-magic`.
- **Impacto:** gate `check-edge-authorization` falha; sem categoria declarada, deploy futuro pode liberar invocação sem autenticação ou requerer dev quando devia ser service.
- **Fix proposto:** declarar cada uma em `supabase/functions/_shared/edge-authz-manifest.ts` com a categoria correta (provavelmente `service` para cron/blurhash e `authenticated` ou `dev` para audit-suite/word-magic — confirmar caso a caso).
