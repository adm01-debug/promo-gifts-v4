Nesta sessão **arquivamos 25 tabelas comprovadamente mortas** (`public` 334 → **309**),

> ⚠️ **Princípio que guiou tudo:** mover uma tabela **em uso** para `archive` quebra a
> produção exatamente como um `DROP` (PostgREST deixa de expô-la; crons/views/funções que
> a referenciam passam a falhar). Por isso **só** movemos objetos com **prova de morte** —
> nada foi movido por “achismo”.

## 2. Metodologia — 5 portões de evidência (todos obrigatórios)

Uma tabela só foi arquivada se passou em **todos** os portões abaixo:

1. **Estrutural** — 0 foreign keys de entrada, 0 views dependentes, não é partição-filha.
2. **Uso histórico** — 0 linhas, **0 escritas** em toda a vida do banco
   (`pg_stat` nunca foi resetado — confirmado: `products` = 113 M scans), < 250 leituras.
3. **Referências no banco** — 0 ocorrências em corpos de função (`pg_proc.prosrc`),
   0 em comandos de cron (`cron.job`), 0 em políticas RLS (`pg_policies`).
4. **Referências no código** — 0 ocorrências como string literal em `src/`,
   `supabase/functions/`, `tests/`, `e2e/` (forma como o cliente Supabase referencia tabelas).
5. **Realtime** — não pertence à publicação `supabase_realtime`.

Tabelas que **falharam** algum portão foram **preservadas** (ver §5, Tier 2).
